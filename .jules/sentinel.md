## 2025-03-03 - [CRITICAL] Fix IDOR Vulnerabilities in Delete Methods
**Vulnerability:** The API had Insecure Direct Object Reference (IDOR) vulnerabilities on the DELETE endpoints for transactions, cards, and accounts. The Supabase repositories `SupabaseTransactionRepository`, `SupabaseCardRepository`, and `SupabaseAccountRepository` were performing deletions using only the provided resource `id`, without verifying if the requested resource belonged to the currently authenticated user (`user_id`). This allowed an authenticated user to delete resources belonging to any other user if they knew or guessed the resource `id`.
**Learning:** This existed because the application architecture segregated repository logic from request context, and the developer trusted that route handler or service layers would validate ownership. However, without ownership checks pushed all the way down to the database query itself, the authorization boundary failed.
**Prevention:** All database mutation methods (UPDATE, DELETE) should always include a `user_id` check alongside the resource identifier. A good architectural rule is to require `userId` in the repository interfaces for any method that acts on user-owned data (`delete(id: string, userId: string)`).
