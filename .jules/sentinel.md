## 2025-03-03 - [CRITICAL] Fix IDOR Vulnerabilities in Delete Methods
**Vulnerability:** The API had Insecure Direct Object Reference (IDOR) vulnerabilities on the DELETE endpoints for transactions, cards, and accounts. The Supabase repositories `SupabaseTransactionRepository`, `SupabaseCardRepository`, and `SupabaseAccountRepository` were performing deletions using only the provided resource `id`, without verifying if the requested resource belonged to the currently authenticated user (`user_id`). This allowed an authenticated user to delete resources belonging to any other user if they knew or guessed the resource `id`.
**Learning:** This existed because the application architecture segregated repository logic from request context, and the developer trusted that route handler or service layers would validate ownership. However, without ownership checks pushed all the way down to the database query itself, the authorization boundary failed.
**Prevention:** All database mutation methods (UPDATE, DELETE) should always include a `user_id` check alongside the resource identifier. A good architectural rule is to require `userId` in the repository interfaces for any method that acts on user-owned data (`delete(id: string, userId: string)`).

## 2025-03-05 - [HIGH] Fix Rate Limiting IP Spoofing Vulnerability
**Vulnerability:** The application was using user-provided HTTP headers (`x-forwarded-for`, `x-real-ip`) unconditionally to identify the client IP for rate limiting on the `/api/*` endpoints. An attacker could trivially bypass rate limits by spoofing these headers, rendering the rate limiter ineffective.
**Learning:** In Hono apps using Node server adapter, using `c.req.header('x-forwarded-for')` or similar headers directly opens up IP spoofing vectors unless the application is explicitly running behind a trusted reverse proxy that guarantees the validity of these headers.
**Prevention:** Always use `@hono/node-server/conninfo` to retrieve `connInfo.remote.address` as the base truth for client IPs. Only trust `x-forwarded-for` or `x-real-ip` if an explicit environment configuration (e.g., `TRUST_PROXY=true`) indicates the app is behind a trusted proxy, and carefully parse the header (e.g., taking the first IP in the `x-forwarded-for` list).

## 2026-03-05 - [HIGH] Unrestricted File Upload Vulnerability in Avatar Upload
**Vulnerability:** The `ProfileService.uploadAvatar` method accepted user-provided `File` objects without validating the file type (MIME type) or file size. This allowed uploading potentially malicious files (e.g., executables, scripts) and exceptionally large files, which could lead to Denial of Service (DoS) or arbitrary code execution if the files were mishandled downstream.
**Learning:** Even if a cloud storage provider (like Supabase) handles the actual upload, the application layer must strictly enforce business constraints—like acceptable MIME types (`image/jpeg`, `image/png`, `image/webp`) and reasonable size limits (e.g., 5MB)—to prevent abuse and reduce attack surface before interacting with external infrastructure.
**Prevention:** Always implement input validation at the application boundary (service layer or route handler) for file uploads. Check both `file.type` against an explicit allowlist and `file.size` against a defined maximum threshold. Returning a standardized `ValidationError` ensures the client is appropriately informed without leaking internal system details.

## 2025-03-08 - [CRITICAL] Fix Path Traversal in Avatar Upload
**Vulnerability:** The `SupabaseProfileRepository.uploadAvatar` method blindly extracted the file extension using `file.name.split('.').pop()`. This allowed an attacker to inject path traversal payloads (e.g., `image.png/../../malicious.txt`) within the file name, potentially writing files outside the intended `avatars/${userId}/` directory within the storage bucket.
**Learning:** Never trust user-provided file names for constructing file paths. Even if the application validates the MIME type in the service layer, a malicious client can send a valid file with a payload-infused file name that bypasses application routing and exploits the storage mechanism.
**Prevention:** Always derive file extensions from the verified MIME type (e.g., mapping `image/jpeg` to `.jpeg`) rather than parsing the user-supplied file name string.

## 2025-03-09 - [CRITICAL] Fix IDOR Vulnerability in Invoice updateStatus
**Vulnerability:** The API had an Insecure Direct Object Reference (IDOR) vulnerability on the `PATCH /api/invoices/:id/paid` endpoint. The Supabase repository `SupabaseInvoiceRepository` was performing status updates using only the provided resource `id`, without verifying if the requested invoice belonged to the currently authenticated user (`user_id`). This allowed an authenticated user to mark any other user's invoice as paid if they knew or guessed the invoice `id`.
**Learning:** Even though the routes had auth middleware to enforce that a user is authenticated, the context of the user making the request must explicitly be passed down to the repository for write operations to enforce isolation and data ownership. This enforces the pattern that all write operations must be bounded by the `user_id`.
**Prevention:** All database mutation methods (UPDATE, DELETE) should always include a `user_id` check alongside the resource identifier, similar to how it was addressed in deletion endpoints previously. Require `userId` in repository interfaces for any method modifying user-owned data (e.g., `updateStatus(id: string, userId: string, status: string)`).

## 2025-03-09 - [WARNING] Build Breakage During IDOR Fix
**Vulnerability:** Not a software vulnerability, but a workflow issue. While executing a security fix for an IDOR vulnerability, an out-of-scope frontend utility file (`formatCurrency.ts`) was accidentally modified by another process or incomplete merge artifact, resulting in duplicated `currencyFormatter` variables and breaking the `apps/web` test suite. The build and linter were not accurately verifying `apps/web` until explicit workspace commands were run.
**Learning:** Security fixes must not break the global build. Duplicated or missing variables in unrelated domain packages can easily be caught by ensuring the verification commands target the entire monorepo (`npx turbo run build`) or by strictly running workspace-specific test commands. Additionally, some tools might auto-import or auto-duplicate definitions when applying AST transforms if not careful.
**Prevention:** Always run full build `npm run build --workspaces` or `npx turbo run build` and tests across the monorepo to ensure side-effects haven't broken the CI pipeline. Ensure all pre-existing CI check failures are explicitly resolved.
